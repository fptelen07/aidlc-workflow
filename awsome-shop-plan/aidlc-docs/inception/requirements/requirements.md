# AWSomeShop 需求文档

## 意图分析摘要

- **用户请求**: 构建名为 AWSomeShop 的内部员工福利电商网站 MVP
- **请求类型**: 新项目（Greenfield）
- **范围估计**: 多组件系统（前端 + 后端 + 数据库）
- **复杂度估计**: 中等
- **商业目标**: 通过 MVP 验证员工积分兑换系统的商业模式

---

## 1. 功能性需求

### 1.1 用户角色

| 角色 | 描述 |
|------|------|
| 员工（Employee） | 普通用户，浏览产品、兑换积分、查看历史 |
| 管理员（Admin） | 管理产品、管理员工积分、系统配置 |

### 1.2 认证与授权

- **FR-AUTH-001**: 系统支持用户名 + 密码的独立认证方式
- **FR-AUTH-002**: 系统根据用户角色（员工/管理员）控制功能访问权限
- **FR-AUTH-003**: 员工和管理员在同一应用中通过角色区分功能入口

### 1.3 产品管理

- **FR-PROD-001**: 管理员可以手动添加、编辑、删除产品信息
- **FR-PROD-002**: 产品信息包含：名称、描述、所需积分、库存数量、产品图片
- **FR-PROD-003**: 系统支持多级产品分类（如：电子产品 > 耳机 > 无线耳机）
- **FR-PROD-004**: 系统支持 200+ 产品规模
- **FR-PROD-005**: 员工可以按分类浏览产品
- **FR-PROD-006**: 员工可以搜索产品（按名称、分类）

### 1.4 积分管理

- **FR-POINT-001**: 系统自动按固定额度定期发放积分（如每月）
- **FR-POINT-002**: 管理员可以手动调整员工积分（发放和扣除）
- **FR-POINT-003**: 员工可以查看当前积分余额
- **FR-POINT-004**: 系统记录所有积分变动历史（发放、扣除、兑换）
- **FR-POINT-005**: 积分变动记录包含：变动类型、变动数量、变动时间、操作人、备注

### 1.5 兑换流程

- **FR-ORDER-001**: 员工可以使用积分兑换产品
- **FR-ORDER-002**: 兑换时系统自动校验积分余额是否充足
- **FR-ORDER-003**: 兑换时系统自动校验产品库存是否充足
- **FR-ORDER-004**: 兑换成功后自动扣除相应积分并减少库存
- **FR-ORDER-005**: 采用线下自取模式，系统仅记录兑换信息
- **FR-ORDER-006**: 员工可以查看自己的兑换历史记录
- **FR-ORDER-007**: 兑换记录包含：产品信息、消耗积分、兑换时间、兑换状态

### 1.6 管理员功能

- **FR-ADMIN-001**: 管理员可以查看所有员工的积分余额
- **FR-ADMIN-002**: 管理员可以查看所有兑换记录
- **FR-ADMIN-003**: 管理员可以管理产品分类（增删改）
- **FR-ADMIN-004**: 管理员可以配置积分自动发放规则（额度、周期）

## 2. 非功能性需求

### 2.1 性能

- **NFR-PERF-001**: 最终目标支持 1000+ 用户，MVP 阶段性能要求可适当放宽
- **NFR-PERF-002**: 页面加载时间 < 3 秒（MVP 标准）
- **NFR-PERF-003**: API 响应时间 < 500ms（常规操作）

### 2.2 安全

- **NFR-SEC-001**: 密码必须加密存储（如 bcrypt）
- **NFR-SEC-002**: API 接口需要认证令牌（如 JWT）保护
- **NFR-SEC-003**: 管理员接口需要额外的角色权限校验
- **NFR-SEC-004**: 防止常见 Web 攻击（XSS、CSRF、SQL 注入）

### 2.3 部署

- **NFR-DEPLOY-001**: 使用 Docker 容器化部署
- **NFR-DEPLOY-002**: 提供 docker-compose 配置，支持一键启动
- **NFR-DEPLOY-003**: 前后端分离部署

### 2.4 数据存储

- **NFR-DATA-001**: 使用 MySQL 作为主数据库
- **NFR-DATA-002**: 数据库需要提供初始化脚本（建表、种子数据）

### 2.5 可维护性

- **NFR-MAINT-001**: 代码需要有清晰的分层架构
- **NFR-MAINT-002**: 提供 API 文档
- **NFR-MAINT-003**: 关键业务逻辑需要有单元测试

### 2.6 技术栈

- **NFR-TECH-001**: 前后端技术框架将在实现阶段确认（用户已有选型）
- **NFR-TECH-002**: 数据库确定使用 MySQL
- **NFR-TECH-003**: 容器化使用 Docker

---

## 3. 约束与假设

### 3.1 约束

- 这是一个 MVP 项目，优先实现核心功能
- 前后端技术栈待实现阶段确认
- 线下自取模式，不涉及物流系统集成

### 3.2 假设

- 员工信息由管理员手动录入或通过初始化脚本导入
- 积分没有有效期限制（MVP 阶段）
- 产品图片使用 URL 引用，不涉及文件上传（MVP 简化）
- 单一货币体系（仅 AWSome 积分，不涉及真实货币）

---

## 4. 范围外（Out of Scope）

- 支付系统集成
- 物流配送跟踪
- 移动端 App
- 邮件/消息通知
- 数据分析和报表
- 多语言支持
- SSO 集成
