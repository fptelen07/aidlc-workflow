## 设计问题

### Q1: 兑换流程的跨服务事务策略
兑换需要同时扣除积分（points-service）和扣减库存（product-service），这是跨服务操作，无法使用本地数据库事务。失败补偿策略：

- A) 顺序调用 + 补偿回滚 — 先扣积分，再扣库存，库存扣减失败则回滚积分；先扣库存，再扣积分，积分扣除失败则恢复库存
- B) 先校验再执行 — 先分别查询积分和库存是否充足（不加锁），校验通过后再依次扣除，失败则补偿
- C) 其他 — 请说明

[Answer]: B

### Q2: 兑换流程的调用顺序
如果选择顺序调用 + 补偿回滚，先扣哪个更合理：

- A) 先扣积分，再扣库存 — 积分回滚比库存恢复更安全（积分是虚拟资产）
- B) 先扣库存，再扣积分 — 库存更稀缺，先锁定库存更合理

[Answer]: A

### Q3: 兑换状态流转
orders 表的 status 字段状态流转设计（参考 infrastructure domain-entities 中定义的 PENDING/READY/COMPLETED/CANCELLED）：

- A) 简化流转 — PENDING（已兑换）→ COMPLETED（已自取/已完成），取消 READY 状态
- B) 完整流转 — PENDING（已兑换）→ READY（可自取）→ COMPLETED（已完成），保留中间状态
- C) 沿用原设计 — PENDING → READY → COMPLETED + CANCELLED（支持取消）

[Answer]: C

### Q4: 兑换取消与退款
如果支持取消兑换（CANCELLED 状态），取消时是否自动退还积分和恢复库存：

- A) 自动退还 — 取消时自动回滚积分和库存
- B) 仅管理员可取消，手动处理退还 — MVP 阶段简化，取消后由管理员手动调整积分
- C) 不支持取消 — MVP 阶段兑换后不可取消

[Answer]: A

### Q5: 跨服务调用超时
order-service 调用 product-service 和 points-service 的超时时间：

- A) 3 秒超时（与 auth-service 跨服务调用策略一致）
- B) 5 秒超时（兑换流程涉及悲观锁，可能需要更长等待）
- C) 其他 — 请指定

[Answer]: A
