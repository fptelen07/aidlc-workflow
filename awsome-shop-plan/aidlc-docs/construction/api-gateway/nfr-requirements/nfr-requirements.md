# Unit 6: api-gateway — 非功能性需求

---

## 1. 安全性

### NFR-GW-SEC-001: JWT 签名安全
- 使用 HS256 算法校验 JWT 签名
- JWT_SECRET 通过环境变量注入，不硬编码
- JWT_SECRET 必须与 auth-service 使用的密钥一致

### NFR-GW-SEC-002: 请求头防伪造
- 转发前必须清除客户端请求中的 X-User-Id 和 X-User-Role
- 防止客户端伪造用户身份绕过认证
- 无论请求是否需要认证，都执行清除

### NFR-GW-SEC-003: 统一认证失败响应
- 所有 JWT 校验失败场景统一返回 401 + GW_001
- 不区分具体失败原因（过期、签名无效、格式错误）
- 防止攻击者通过不同错误信息推断系统内部状态

### NFR-GW-SEC-004: 输入校验
- 校验 Authorization 请求头格式（Bearer 前缀）
- 校验 JWT payload 必要字段完整性（userId、username、role）
- 校验 role 值合法性（仅 EMPLOYEE 或 ADMIN）

### NFR-GW-SEC-005: 内部接口隔离
- 网关不路由 /api/internal/* 请求
- 内部接口仅在 Docker 内部网络中服务间直接调用
- 如果客户端请求 /api/internal/* 路径，返回 404

## 2. 性能

### NFR-GW-PERF-001: 网关自身处理开销
- JWT 校验 + 路由匹配 + 请求头处理的 P95 开销 ≤ 50ms
- 不含下游微服务响应时间
- JWT 校验为纯计算操作（HMAC-SHA256），无 I/O

### NFR-GW-PERF-002: 转发超时
- 连接超时：1 秒
- 读取超时：2 秒
- 总超时：3 秒
- 与 order-service 跨服务调用超时配置一致

### NFR-GW-PERF-003: 无状态设计
- 网关不维护会话状态
- 每个请求独立处理，不依赖前序请求
- 支持水平扩展（虽然 MVP 阶段单实例）

---

## 3. 可靠性

### NFR-GW-REL-001: 下游不可达处理
- 下游微服务连接失败 → 返回 GW_003 (502)
- 下游微服务响应超时 → 返回 GW_004 (504)
- 不自动重试（网关层不做重试，由调用方或下游服务自行处理）

### NFR-GW-REL-002: 下游错误透传
- 下游微服务返回的 4xx/5xx 错误原样透传给客户端
- 网关不修改下游的 HTTP 状态码和响应体
- 仅网关自身产生的错误使用 GW_ 错误码

### NFR-GW-REL-003: 启动依赖
- 网关启动前需要确认 JWT_SECRET 环境变量已配置
- JWT_SECRET 为空或未配置 → 网关启动失败（快速失败）

---

## 4. 可维护性

### NFR-GW-MAINT-001: 统一错误响应格式
- 网关层错误遵循统一格式：`{ code, message, data }`
- 错误码前缀：GW_
- 与其他微服务保持一致

### NFR-GW-MAINT-002: 日志规范
- 遵循后端技术框架的日志规范
- 认证失败必须记录日志（包含请求路径、客户端 IP）
- 权限校验失败必须记录日志
- 下游不可达/超时必须记录 WARN 级别日志

### NFR-GW-MAINT-003: 配置外部化
- JWT_SECRET 通过环境变量注入
- 所有下游服务地址通过环境变量配置（AUTH_SERVICE_URL 等）
- 超时时间可通过配置调整

### NFR-GW-MAINT-004: 路由配置可维护
- 路由规则集中定义，便于新增或修改
- 权限规则集中定义，便于调整端点访问级别
- 新增微服务时只需添加路由规则和环境变量

---

## 5. 可测试性

### NFR-GW-TEST-001: 认证逻辑可测试
- JWT 校验逻辑可通过单元测试验证
- 可构造不同的 JWT（过期、签名无效、缺少字段）测试各种失败场景

### NFR-GW-TEST-002: 路由逻辑可测试
- 路由匹配逻辑可通过单元测试验证
- 可验证不同 URL 路径匹配到正确的目标服务

### NFR-GW-TEST-003: 权限逻辑可测试
- 权限校验逻辑可通过单元测试验证
- 可验证公开端点、已认证端点、管理员端点的访问控制
