# Unit 5: order-service — NFR 设计模式

---

## 模式 1: 跨服务调用重试与容错

### 适用场景
- order-service 调用 product-service（查询产品、扣减/恢复库存）
- order-service 调用 points-service（查询余额、扣除/回滚积分）

### 设计
```
跨服务调用
  → 设置超时: 连接 1s + 读取 2s = 总计 3s
  → 发起 HTTP 请求
  → 成功 → 返回结果
  → 超时/网络异常 → 自动重试 1 次（立即重试，无延迟）
      → 成功 → 返回结果
      → 仍失败 → 根据调用阶段执行补偿逻辑
```

### 重试安全矩阵

| 接口 | 是否可重试 | 幂等性保证 |
|------|-----------|-----------|
| GET /api/internal/products/{id} | ✅ 安全 | 只读操作 |
| GET /api/internal/points/balance/{userId} | ✅ 安全 | 只读操作 |
| POST /api/internal/points/deduct | ✅ 安全 | orderId 关联去重 |
| POST /api/internal/products/deduct-stock | ⚠️ 需确认 | 悲观锁保证不超卖，但需确保幂等 |
| POST /api/internal/points/rollback | ✅ 安全 | 重复回滚返回 POINTS_006 |
| POST /api/internal/products/restore-stock | ⚠️ 需确认 | 需确保重复恢复不多加库存 |

### 关键点
- 仅对超时和网络异常重试，业务错误（4xx）不重试
- 重试前不需要等待（立即重试），适合 MVP 阶段
- 被调用方的交易类接口必须保证幂等性（用户明确要求）
- 重试次数硬编码为 1 次，后续可通过配置调整

### 覆盖 NFR
- NFR-ORD-REL-001（跨服务调用重试）
- NFR-ORD-REL-002（幂等性依赖）
- NFR-ORD-REL-004（跨服务调用超时）

## 模式 2: 顺序扣除与补偿回滚

### 适用场景
- 创建兑换订单（核心流程）

### 设计
```
创建兑换订单
  │
  ├── 阶段一：预校验（不加锁，快速拦截）
  │     ├── 查询产品信息 → 失败则直接返回错误
  │     └── 查询积分余额 → 失败则直接返回错误
  │
  ├── 阶段二：创建订单记录（status=PENDING）
  │     └── 获得 orderId，用于后续关联
  │
  ├── 阶段三：顺序扣除（先积分后库存）
  │     ├── 扣除积分（传入 orderId）
  │     │     └── 失败 → 删除订单记录，返回错误
  │     └── 扣减库存
  │           └── 失败 → 回滚积分 + 删除订单记录，返回错误
  │
  └── 返回 OrderResponse
```

### 补偿决策树
```
扣除积分失败
  └── 删除订单记录 → 返回错误（无需回滚积分，因为积分未扣成功）

扣减库存失败
  ├── 回滚积分
  │     ├── 成功 → 删除订单记录 → 返回错误
  │     └── 失败 → 记录 ERROR 日志（含 orderId、transactionId）
  │               → 删除订单记录 → 返回错误
  └── 注意：订单记录必须删除，避免产生无效的 PENDING 订单
```

### 关键点
- 先扣积分后扣库存：积分是虚拟资产，回滚更安全可靠
- 预校验阶段不加锁，减少锁竞争
- 预校验通过但执行时条件已变：由被调用方的悲观锁保证一致性
- 补偿采用"最大努力"策略，失败记录日志由管理员处理

### 覆盖 NFR
- NFR-ORD-REL-003（补偿一致性）

## 模式 3: 统一错误处理

### 适用场景
- 所有 API 端点（员工端 + 管理员端）

### 设计
```json
{
  "code": "ORDER_001",
  "message": "产品不存在",
  "data": null
}
```

### 错误分类

| HTTP 状态码 | 场景 |
|------------|------|
| 400 | 参数校验失败、产品已下架、库存不足、积分不足、非法状态变更 |
| 403 | 无权查看他人订单 |
| 404 | 产品不存在、订单不存在 |
| 500 | 跨服务调用失败、补偿处理异常 |

### 跨服务错误转换
```
被调用方返回错误
  → 解析错误码
  → 转换为 order-service 自身的错误码
  → 例: product-service 返回 "产品不存在" → ORDER_001
  → 例: points-service 返回 "余额不足" → ORDER_005
  → 超时/网络异常 → ORDER_008（兑换处理失败，请稍后重试）
```

### 关键点
- 业务异常返回 4xx，系统异常返回 5xx
- 跨服务错误需转换为本服务错误码，不暴露内部服务细节
- 补偿失败使用 ORDER_010，订单仍标记为 CANCELLED
- 与其他微服务保持一致的错误响应格式

### 覆盖 NFR
- NFR-ORD-MAINT-001（统一错误响应格式）

## 模式 4: 分层输入校验

### 适用场景
- 所有接收外部输入的 API 端点

### 设计
```
请求进入
  → 第1层: 框架级校验（参数类型、必填、范围）
      → productId > 0
      → status 为合法枚举值
      → 分页参数范围校验
  → 第2层: 业务逻辑校验（跨服务查询验证）
      → 产品存在性、状态、库存
      → 积分余额充足性
      → 订单存在性、归属校验
      → 状态流转合法性
  → 执行业务操作
```

### 关键点
- 第1层校验失败直接返回 400，不发起跨服务调用
- 第2层校验中的跨服务查询属于预校验阶段（不加锁）
- 订单归属校验在第2层执行，防止越权访问

### 覆盖 NFR
- NFR-ORD-SEC-001（订单归属校验）
- NFR-ORD-SEC-002（输入校验）

## 模式 5: 产品快照冗余

### 适用场景
- 创建兑换订单时保存产品信息快照

### 设计
```
创建订单
  → 从 product-service 查询产品信息
  → 将 productName、productImageUrl 冗余存储到 orders 表
  → 后续查询订单时直接返回冗余字段，不跨服务查询
```

### 关键点
- 快照在创建订单时一次性写入，后续不更新
- 产品被修改或删除不影响历史兑换记录的展示
- 管理员按产品名称搜索使用冗余字段，不跨服务查询
- 减少查询阶段的跨服务依赖，提升查询性能

### 覆盖 NFR
- NFR-ORD-REL-005（产品快照完整性）
- NFR-ORD-PERF-002（查询接口响应时间）

## 模式 6: 取消自动退还

### 适用场景
- 管理员将订单状态更新为 CANCELLED

### 设计
```
管理员取消订单
  │
  ├── 1. 校验状态流转合法性
  │     └── 仅 PENDING/READY → CANCELLED 合法
  │
  ├── 2. 更新订单状态为 CANCELLED
  │     └── 先更新状态，确保取消操作不被阻塞
  │
  ├── 3. 回滚积分（最大努力）
  │     └── 使用 orders.points_transaction_id 调用回滚接口
  │         ├── 成功 → 继续
  │         └── 失败 → 记录 ERROR 日志（含 orderId、transactionId）
  │
  ├── 4. 恢复库存（最大努力）
  │     └── 调用 product-service 恢复接口
  │         ├── 成功 → 继续
  │         └── 失败 → 记录 ERROR 日志（含 orderId、productId）
  │
  └── 5. 返回更新后的 OrderResponse
```

### 关键点
- 先更新状态再执行退还，退还失败不阻塞状态更新
- 积分回滚使用创建订单时保存的 points_transaction_id
- 退还失败记录 ERROR 级别日志，包含足够的排查信息
- MVP 阶段由管理员定期检查日志处理异常

### 覆盖 NFR
- NFR-ORD-REL-003（补偿一致性）
- NFR-ORD-MAINT-002（日志规范）

## 模式 7: 配置外部化

### 适用场景
- 数据库连接、跨服务调用地址、超时配置

### 设计

| 配置项 | 环境变量 | 默认值 | 说明 |
|--------|---------|--------|------|
| 数据库 URL | DB_HOST, DB_PORT, DB_NAME | mysql:3306/order_db | 数据库连接 |
| 数据库用户 | DB_USERNAME | — | 数据库用户名 |
| 数据库密码 | DB_PASSWORD | — | 数据库密码 |
| 产品服务地址 | PRODUCT_SERVICE_URL | http://product-service:8002 | product-service 基础 URL |
| 积分服务地址 | POINTS_SERVICE_URL | http://points-service:8003 | points-service 基础 URL |
| 连接超时 | CONNECT_TIMEOUT | 1000 | 跨服务连接超时（ms） |
| 读取超时 | READ_TIMEOUT | 2000 | 跨服务读取超时（ms） |

### 关键点
- 所有外部依赖地址通过环境变量注入
- 超时时间可通过配置调整，无需修改代码
- Docker Compose 中通过 environment 配置

### 覆盖 NFR
- NFR-ORD-MAINT-003（配置外部化）
