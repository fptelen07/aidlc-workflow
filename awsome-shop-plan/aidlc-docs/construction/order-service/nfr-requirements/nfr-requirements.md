# Unit 5: order-service — 非功能性需求

---

## 1. 安全性

### NFR-ORD-SEC-001: 订单归属校验
- 员工查看兑换详情时，必须校验订单 userId 与当前用户一致
- 不允许查看他人订单，返回 403

### NFR-ORD-SEC-002: 输入校验
- 所有外部输入必须经过参数校验
- 防止非法数据进入业务逻辑层

### NFR-ORD-SEC-003: 内部接口依赖安全
- order-service 调用的内部接口（product-service、points-service）均在 Docker 内部网络中
- 不经过 API 网关，服务间信任

---

## 2. 性能

### NFR-ORD-PERF-001: 兑换流程响应时间
- 创建兑换订单整体 P95 响应时间 ≤ 2 秒
- 包含预校验（2 次跨服务查询）+ 执行扣除（2 次跨服务写入）+ 本地订单创建
- 考虑悲观锁等待和网络波动

### NFR-ORD-PERF-002: 查询接口响应时间
- 订单查询类接口（历史列表、详情、管理员列表）P95 响应时间 ≤ 200ms
- 与其他微服务查询接口性能目标一致

### NFR-ORD-PERF-003: 分页查询性能
- 分页查询使用数据库级别分页（LIMIT/OFFSET）
- 默认每页 20 条，最大 100 条
- 利用 orders 表的 user_id + created_at 索引优化查询

---

## 3. 可靠性

### NFR-ORD-REL-001: 跨服务调用重试
- 调用 product-service 和 points-service 时，超时或网络异常自动重试 1 次
- 重试间隔：立即重试（无延迟）
- 仍失败则执行补偿逻辑
- **前提条件**：被调用的交易类接口必须保证幂等性

### NFR-ORD-REL-002: 幂等性依赖
- 积分扣除接口（points-service）：通过 orderId 关联保证同一订单不会重复扣除
- 库存扣减接口（product-service）：通过悲观锁保证不会超卖
- 积分回滚接口（points-service）：已设计为幂等（重复回滚返回错误码 POINTS_006）
- 库存恢复接口（product-service）：需确保幂等性（重复恢复不会多加库存）

### NFR-ORD-REL-003: 补偿一致性
- 兑换失败时的补偿采用"最大努力"策略
- 补偿操作（积分回滚、库存恢复）失败时记录错误日志
- MVP 阶段由管理员定期检查日志处理异常情况

### NFR-ORD-REL-004: 跨服务调用超时
- 所有跨服务调用超时时间：3 秒（连接超时 1 秒 + 读取超时 2 秒）
- 超时后触发重试（1 次），仍失败则执行补偿

### NFR-ORD-REL-005: 产品快照完整性
- 创建订单时冗余存储产品名称和图片 URL
- 确保兑换历史不受产品后续修改/删除影响

---

## 4. 可维护性

### NFR-ORD-MAINT-001: 统一错误响应格式
- 所有错误响应遵循统一格式：`{ code, message, data }`
- 错误码前缀：ORDER_
- 与其他微服务保持一致

### NFR-ORD-MAINT-002: 日志规范
- 遵循后端技术框架的日志规范
- 兑换流程关键步骤必须记录日志（预校验结果、扣除结果、补偿操作）
- 补偿失败必须记录 ERROR 级别日志，包含订单 ID、失败原因、需人工处理的信息

### NFR-ORD-MAINT-003: 配置外部化
- 数据库连接信息通过环境变量注入
- 跨服务调用地址通过环境变量配置（PRODUCT_SERVICE_URL、POINTS_SERVICE_URL）
- 超时时间可通过配置调整

---

## 5. 可测试性

### NFR-ORD-TEST-001: 接口可测试
- 所有 API 端点可通过 HTTP 客户端独立测试
- 跨服务调用可通过 Mock 替换进行单元测试

### NFR-ORD-TEST-002: 补偿逻辑可测试
- 补偿回滚逻辑可独立触发测试
- 可模拟跨服务调用失败场景验证补偿行为
