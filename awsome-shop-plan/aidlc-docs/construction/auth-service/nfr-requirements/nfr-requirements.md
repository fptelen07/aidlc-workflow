# Unit 2: auth-service — NFR 需求

---

## 安全性

### NFR-AUTH-001: 密码加密存储
- 所有用户密码必须使用 bcrypt 加密后存储
- bcrypt cost factor = 10
- 明文密码不得出现在日志、响应、数据库中

### NFR-AUTH-002: JWT 令牌安全
- 签名算法：HS256
- JWT_SECRET 通过环境变量注入，不得硬编码
- 令牌 payload 不包含敏感信息（不含密码、不含完整个人信息）
- 令牌过期时间可配置（默认 24 小时）

### NFR-AUTH-003: 登录信息脱敏
- 登录失败时统一返回"用户名或密码错误"
- 不向客户端暴露用户是否存在的信息
- 防止用户名枚举攻击

### NFR-AUTH-004: 输入校验
- 所有 API 输入参数必须进行校验
- 防止 SQL 注入（参数化查询）
- 防止 XSS（输出编码）
- 密码长度上限 128 位，防止 bcrypt DoS

---

## 性能

### NFR-AUTH-005: API 响应时间
- P95 响应时间目标：200ms 以内（不含 bcrypt 操作）
- 注册和登录接口因 bcrypt 操作，P95 目标放宽至 500ms
- MVP 阶段不做并发优化，单线程处理 bcrypt

### NFR-AUTH-006: 分页查询性能
- 用户列表查询支持分页，单页最大 100 条
- keyword 搜索使用数据库索引优化（username、employeeId 已有索引）

---

## 可靠性

### NFR-AUTH-007: 跨服务调用容错
- 调用 points-service 超时时间：3 秒
- 调用失败时降级处理：注册仍然成功，记录错误日志
- 积分初始化可在后续首次查询时补偿

### NFR-AUTH-008: 数据一致性
- 用户名和工号的唯一性由数据库 UNIQUE 约束保证
- 并发注册相同用户名/工号时，数据库约束兜底

---

## 可维护性

### NFR-AUTH-009: 统一错误响应
- 所有错误使用统一格式：`{ code, message, data }`
- 错误码前缀：AUTH_
- 业务异常返回 4xx，系统异常返回 5xx

### NFR-AUTH-010: 日志规范
- 遵循后端统一技术框架的日志规范
- 关键操作记录日志：注册成功/失败、登录成功/失败、账号状态变更
- 日志不得包含密码明文

### NFR-AUTH-011: 配置外部化
- 所有配置通过环境变量注入
- 必需配置：DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD, JWT_SECRET, JWT_EXPIRATION
- 可选配置：POINTS_SERVICE_URL（默认 http://points-service:8003）

---

## 可测试性

### NFR-AUTH-012: 单元测试
- 核心业务逻辑（注册校验、密码加密、JWT 生成）需有单元测试覆盖
- 测试框架随后端技术框架确定

### NFR-AUTH-013: API 测试
- 所有 API 端点需有集成测试覆盖
- 测试数据使用独立的测试数据库或内存数据库
