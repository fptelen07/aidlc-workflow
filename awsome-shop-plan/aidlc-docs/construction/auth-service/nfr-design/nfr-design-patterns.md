# Unit 2: auth-service — NFR 设计模式

---

## 1. 密码安全模式

### 模式：bcrypt 单向加密 + 盐值内嵌

```
注册流程:
  明文密码 → bcrypt.hash(password, costFactor=10) → 加密字符串 → 存储到 DB

登录流程:
  明文密码 + DB中的加密字符串 → bcrypt.verify(password, hash) → true/false
```

设计要点：
- bcrypt 自动生成随机盐值并嵌入哈希结果中，无需单独存储盐值
- cost factor = 10，单次加密约 100ms，安全性与性能的平衡点
- 密码长度上限 128 位，防止超长密码导致 bcrypt DoS

## 2. 无状态认证模式

### 模式：JWT 令牌 + API 网关校验

```
                    生成令牌                    校验令牌
auth-service ──────────────→ 客户端 ──────────────→ api-gateway
  (HS256签名)                (localStorage)         (HS256验签)
                                                       │
                                                       ▼
                                                  转发请求到微服务
                                                  (附加 X-User-Id, X-User-Role)
```

设计要点：
- auth-service 仅负责生成令牌，不负责校验
- api-gateway 负责所有请求的令牌校验
- 两者共享 JWT_SECRET 环境变量
- 令牌 payload：userId、username、role、iat、exp
- 退出登录：前端清除令牌，后端无操作

## 3. 跨服务调用容错模式

### 模式：超时 + 降级

```
auth-service 注册流程:
  ├── 1. 创建用户 (本地DB) ── 必须成功
  └── 2. 调用 points-service 初始化积分
        ├── 成功 → 完成
        ├── 超时(3s) → 降级：记录日志，注册仍成功
        └── 异常 → 降级：记录日志，注册仍成功

补偿机制:
  points-service 在首次查询积分时，若余额记录不存在，自动创建（balance=0）
```

设计要点：
- HTTP 调用超时：3 秒（连接超时 1 秒 + 读取超时 2 秒）
- 降级策略：调用失败不影响主流程（注册成功）
- 补偿策略：points-service 懒初始化兜底
- 内部接口路径：POST http://points-service:8080/api/internal/points/init

## 4. 统一错误处理模式

### 模式：错误码 + 统一响应格式

```json
// 成功响应
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": { ... }
}

// 错误响应
{
  "code": "AUTH_001",
  "message": "用户名已存在",
  "data": null
}
```

错误码映射：

| 错误码 | HTTP 状态码 | 场景 |
|--------|-----------|------|
| AUTH_001 | 409 | 用户名已存在 |
| AUTH_002 | 409 | 工号已存在 |
| AUTH_003 | 401 | 用户名或密码错误 |
| AUTH_004 | 404 | 用户不存在 |
| AUTH_005 | 400 | 参数校验失败 |
| AUTH_006 | 403 | 账号已被禁用 |
| AUTH_007 | 400 | 不能禁用自己的账号 |

设计要点：
- 全局异常处理器捕获业务异常，转换为统一格式
- 未预期异常返回 500 + 通用错误信息，不暴露内部细节
- 参数校验失败返回具体字段错误信息

## 5. 输入校验模式

### 模式：分层校验

```
第1层：框架层参数校验（注解/装饰器）
  ├── 非空校验
  ├── 长度校验
  └── 格式校验（正则）

第2层：业务层逻辑校验
  ├── 唯一性校验（查数据库）
  ├── 状态校验（账号是否禁用）
  └── 权限校验（不能禁用自己）

第3层：数据库层约束兜底
  ├── UNIQUE 约束
  └── NOT NULL 约束
```

设计要点：
- 尽早失败：框架层校验在进入业务逻辑前拦截无效请求
- 数据库约束作为最后防线，处理并发场景
- 校验错误返回 AUTH_005 + 具体字段信息
