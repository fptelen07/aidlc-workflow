# Unit 4: points-service — 非功能性需求

---

## 1. 安全性

### NFR-PTS-SEC-001: 内部接口网络隔离
- 内部接口（/api/internal/*）仅在 Docker 内部网络中可访问
- 不经过 API 网关，不需要 JWT 认证
- 依赖 Docker 网络隔离保证安全

### NFR-PTS-SEC-002: 管理员操作审计
- 管理员手动调整积分时，必须记录操作人 ID（operatorId）
- 积分变动流水表（point_transactions）作为审计日志，不可删除、不可修改

### NFR-PTS-SEC-003: 输入校验
- 所有外部输入必须经过参数校验
- 防止非法数据进入业务逻辑层
- 校验规则参见功能设计 business-rules.md

## 2. 性能

### NFR-PTS-PERF-001: API 响应时间
- 所有 API 接口（含内部接口）P95 响应时间 ≤ 200ms
- 与 auth-service 性能目标一致

### NFR-PTS-PERF-002: 悲观锁超时
- 积分扣除/调整操作的悲观锁等待超时：5 秒
- 超时后快速失败，返回错误信息
- 与 product-service 库存悲观锁策略一致
- 实现方式：事务级别设置 `SET innodb_lock_wait_timeout = 5`

### NFR-PTS-PERF-003: 分页查询性能
- 分页查询使用数据库级别分页（LIMIT/OFFSET）
- 默认每页 20 条，最大 100 条
- 利用 point_transactions 表的 user_id + created_at 索引优化查询

### NFR-PTS-PERF-004: 定时任务执行效率
- 积分自动发放采用逐条处理、独立事务模式
- 每条发放操作为简单的 UPDATE + INSERT，预期单条 < 50ms
- 1000 用户发放预计总耗时 < 60 秒

## 3. 可靠性

### NFR-PTS-REL-001: 积分变动事务一致性
- 余额更新和流水记录创建必须在同一数据库事务中
- 事务失败时完整回滚，不允许出现余额变动但无流水记录的情况

### NFR-PTS-REL-002: 定时任务可靠性 — 发放批次记录
- 新增发放批次表（distribution_batches），记录每次发放的执行状态
- 批次状态：RUNNING → COMPLETED / FAILED
- 服务重启后检查是否有 RUNNING 状态的未完成批次，支持补发
- 批次记录包含：批次ID、发放额度、总人数、成功数、失败数、开始时间、完成时间、状态

### NFR-PTS-REL-003: 定时任务单条隔离
- 批量发放时，每位用户的发放操作为独立事务
- 单个用户发放失败不影响其他用户
- 失败用户记录在批次明细中，可由管理员后续处理

### NFR-PTS-REL-004: 幂等性保证
- 积分初始化接口（/api/internal/points/init）为幂等操作
- 兑换回滚接口通过唯一性校验防止重复回滚

### NFR-PTS-REL-005: 跨服务调用容错
- order-service 调用积分扣除/回滚接口时，调用方负责超时和重试策略
- points-service 自身保证接口的幂等性和事务一致性

## 4. 可维护性

### NFR-PTS-MAINT-001: 统一错误响应格式
- 所有错误响应遵循统一格式：`{ code, message, data }`
- 错误码前缀：POINTS_
- 与其他微服务保持一致的错误处理模式

### NFR-PTS-MAINT-002: 日志规范
- 遵循后端技术框架的日志规范（延后到实现阶段确认）
- 定时任务执行结果必须记录日志（发放总人数、成功数、失败数）
- 积分变动操作记录关键业务日志

### NFR-PTS-MAINT-003: 配置外部化
- 数据库连接信息通过环境变量注入
- 定时任务 cron 表达式硬编码（固定每月 1 日凌晨 2:00）
- 发放额度通过 system_configs 表动态配置

---

## 5. 可测试性

### NFR-PTS-TEST-001: 接口可测试
- 所有 API 端点可通过 HTTP 客户端独立测试
- 内部接口可在 Docker 网络内直接调用测试

### NFR-PTS-TEST-002: 定时任务可测试
- 定时任务的核心逻辑（发放方法）可独立调用测试
- 不依赖 cron 触发器即可验证发放逻辑
