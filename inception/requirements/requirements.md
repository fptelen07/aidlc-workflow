# AWSome Shop 需求文档

## 意图分析

- **用户请求**: 开发所有未完成业务功能（3 人团队）
- **请求类型**: 新功能开发（New Feature）
- **范围估计**: 系统级（System-wide）— 4 个后端服务 + 前端 + API 对接
- **复杂度估计**: 复杂（Complex）— 多服务、多页面、跨服务交互

---

## 一、功能需求

### FR-001: 用户认证服务 (Auth Service)

#### FR-001.1: 用户注册
- 员工通过用户名 + 密码 + 显示名称自助注册
- 注册后自动分配 `employee` 角色
- 初始积分为 0，由管理员后续发放
- 用户名唯一性校验
- 密码强度要求：最少 6 位

#### FR-001.2: 用户登录
- 用户名 + 密码登录
- 成功返回 JWT access token + refresh token
- access token 有效期 2 小时，refresh token 有效期 7 天
- 返回用户信息（username, displayName, role, points, avatar）

#### FR-001.3: Token 刷新
- 使用 refresh token 获取新的 access token
- refresh token 过期后需重新登录

#### FR-001.4: Token 验证（内部接口）
- Gateway 调用的内部接口：验证 token 有效性，返回 operatorId
- 路径：POST /api/v1/internal/auth/validate

#### FR-001.5: 获取当前用户信息
- 根据 token 返回当前用户完整信息（含积分余额）

#### FR-001.6: 密码加密
- 使用 BCrypt 加密存储密码

---

### FR-002: 商品服务 (Product Service)

#### FR-002.1: 商品管理（管理端）
- 创建商品：名称、描述、分类 ID、积分价格、库存数量、图片 URL、状态（上架/下架）
- 编辑商品：修改所有字段
- 删除商品：逻辑删除
- 商品列表：分页、按名称搜索、按分类筛选、按状态筛选
- 上架/下架切换

#### FR-002.2: 分类管理（管理端）
- 创建分类：名称、排序权重
- 编辑分类：修改名称和排序
- 删除分类：逻辑删除（有商品关联时禁止删除）
- 分类列表：全量查询

#### FR-002.3: 商品浏览（员工端）
- 商品列表：分页、按分类筛选、按名称搜索
- 仅展示已上架且库存 > 0 的商品
- 商品详情页：完整信息（名称、描述、图片、积分价格、库存、分类）

#### FR-002.4: 默认分类（Flyway 初始化）
- 数码电子、生活家居、美食餐饮、礼品卡券、办公用品

---

### FR-003: 订单服务 (Order Service)

#### FR-003.1: 创建兑换订单（员工端）
- 员工选择商品 → 创建订单（状态：待处理 pending）
- 校验：商品存在且上架、库存充足、用户积分充足
- 创建时不扣减积分（管理员确认后扣减）
- 记录：用户 ID、商品 ID、商品名称、积分数量、创建时间

#### FR-003.2: 订单管理（管理端）
- 订单列表：分页、按状态筛选、按用户搜索
- 确认订单：pending → completed，触发积分扣减和库存扣减
- 拒绝订单：pending → rejected（不扣减积分）
- 查看订单详情

#### FR-003.3: 兑换记录（员工端）
- 查看自己的兑换订单列表：分页、按状态筛选
- 查看订单详情

#### FR-003.4: 订单状态流转
```
pending（待处理）→ completed（已完成）  [管理员确认]
pending（待处理）→ rejected（已拒绝）   [管理员拒绝]
```

---

### FR-004: 积分服务 (Points Service)

#### FR-004.1: 积分余额
- 查询用户当前积分余额
- 积分余额不允许为负数

#### FR-004.2: 积分发放（管理端）
- 管理员给单个用户发放积分（指定数量和原因）
- 批量发放：给多个用户同时发放积分
- 记录积分变动流水

#### FR-004.3: 积分扣减
- 兑换订单确认时系统自动扣减积分
- 扣减前校验余额充足
- 记录积分变动流水

#### FR-004.4: 积分变动历史
- 员工查看自己的积分变动记录：分页、按类型筛选（发放/扣减）
- 管理员查看所有用户的积分变动记录

#### FR-004.5: 积分过期
- 积分设置有效期（可配置，默认 12 个月）
- 过期积分自动失效
- 积分变动历史中标记过期记录

#### FR-004.6: 积分统计（管理端）
- 积分流通量统计（月度发放总量、扣减总量）
- 用户积分排行

#### FR-004.7: 发放规则配置（管理端）
- 配置积分发放规则（如：新员工注册奖励、月度固定发放等）
- 规则启用/禁用

## 二、非功能需求

### NFR-001: 性能
- API 响应时间 < 500ms（P95）
- 商品列表页加载 < 1s
- 支持 100 并发用户

### NFR-002: 安全
- 密码 BCrypt 加密存储
- JWT token 签名验证
- 所有非公开 API 需要认证
- 防止 SQL 注入（MyBatis-Plus 参数化查询）
- CORS 配置限制

### NFR-003: 可维护性
- 遵循现有 DDD 六边形架构分层
- 保持 POST-only API 设计规范一致性
- 代码注释和 Swagger API 文档

### NFR-004: 测试
- 领域服务和应用服务单元测试
- 仓储层集成测试（TestContainers + MySQL + Redis）
- 测试覆盖率目标：核心业务逻辑 > 70%

### NFR-005: 国际化
- 前端所有新页面支持中文/英文切换
- 后端错误消息使用错误码，前端负责翻译

### NFR-006: 数据一致性
- 服务间同步 HTTP 调用（经网关）
- 订单确认时的积分扣减和库存扣减需保证一致性
- 使用数据库事务保证单服务内一致性

## 三、架构决策记录

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 认证方式 | 纯 JWT (access + refresh token) | 无状态，微服务友好，实现简单 |
| 注册流程 | 简单自助注册 | 内部系统，无需审批流程 |
| 商品模型 | 精简版 | MVP 阶段，后续可扩展 |
| 兑换流程 | 两步（员工兑换 → 管理员确认） | 平衡用户体验和管理控制 |
| 积分操作 | 标准（含批量发放、过期、规则配置） | 满足企业积分管理需求 |
| 服务间通信 | 同步 HTTP 经网关 | 最简单，适合当前规模 |
| API 设计 | POST-only | 与现有脚手架一致 |
| 管理端页面 | 标准（搜索/筛选/批量/图片上传） | 满足日常管理需求 |
| 数据初始化 | Flyway SQL | 可重复、版本化 |
| 测试策略 | 单元 + 集成测试 | 平衡质量和交付速度 |
| 安全扩展 | 不启用 | 非生产级，PoC 阶段 |
| 属性测试 | 部分启用（纯函数+序列化） | 适度覆盖核心逻辑 |
| 技术债务 | 全部修复 | 趁早解决，避免后续累积 |
