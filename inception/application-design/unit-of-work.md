# 工作单元定义

## 分解策略
- **分解方式**: 按微服务边界，每个服务为一个独立工作单元
- **团队分工**: 按服务分工，每人负责 1-2 个后端服务全栈开发
- **开发顺序**: 按依赖关系排列，阻塞路径优先

---

## Unit 1: 技术债务修复 (Tech Debt)
- **负责人**: 全员协作（开发前统一处理）
- **范围**: 统一 Result 类、修复 Gateway ThreadLocal、实现 AOP、清理占位代码
- **交付物**: 干净的代码库，统一的框架基础
- **前置依赖**: 无
- **预估工作量**: 小（0.5 天）

## Unit 2: Auth Service (认证服务)
- **负责人**: 开发者 A
- **范围**: 用户注册/登录、JWT 生成/验证/刷新、密码加密、Flyway 迁移、种子数据
- **交付物**: 可运行的认证服务，Gateway 可调用 token 验证接口
- **前置依赖**: Unit 1 (技术债务)
- **预估工作量**: 中（2 天）

## Unit 3: Product Service (商品服务)
- **负责人**: 开发者 B
- **范围**: 商品 CRUD、分类 CRUD、商品浏览/详情、上下架、库存扣减、Flyway 迁移、种子数据
- **交付物**: 可运行的商品服务，提供商品查询和库存管理 API
- **前置依赖**: Unit 1 (技术债务)
- **预估工作量**: 中（2 天）

## Unit 4: Points Service (积分服务)
- **负责人**: 开发者 C
- **范围**: 积分账户、发放/扣减、批量发放、变动历史、过期处理、规则配置、统计、Flyway 迁移
- **交付物**: 可运行的积分服务，提供积分管理 API
- **前置依赖**: Unit 1 (技术债务)
- **预估工作量**: 中（2.5 天）

## Unit 5: Order Service (订单服务)
- **负责人**: 开发者 A（Auth 完成后接手）
- **范围**: 兑换订单创建、确认/拒绝、状态流转、跨服务调用（Product + Points）、Flyway 迁移
- **交付物**: 可运行的订单服务，完成兑换业务闭环
- **前置依赖**: Unit 2 (Auth), Unit 3 (Product), Unit 4 (Points)
- **预估工作量**: 中（2 天）

## Unit 6: Frontend (前端)
- **负责人**: 全员协作（后端完成后）
- **范围**: 9 个新页面、7 个 API Service 模块、Mock 替换、注册页、商品详情页
- **交付物**: 完整前端，对接所有后端 API
- **前置依赖**: Unit 2, 3, 4, 5 (所有后端服务)
- **预估工作量**: 大（3 天）
- **子任务分配**:
  - 开发者 A: 注册页、登录对接、员工兑换记录页、管理端订单页
  - 开发者 B: 商品详情页、管理端商品页、管理端分类页、ShopHome 对接
  - 开发者 C: 员工积分中心页、管理端积分页、管理端用户页、Dashboard 对接

---

## 开发时间线

```
Day 1 (全员):     Unit 1 技术债务 ──────────────────────┐
                                                         │
Day 2-3:          Unit 2 Auth (A) ───┐                   │
                  Unit 3 Product (B) ─┤  可并行           │
                  Unit 4 Points (C) ──┘                   │
                                                         │
Day 4-5:          Unit 5 Order (A) ──── 依赖 2,3,4       │
                  (B,C 可提前开始 Unit 6 前端基础)         │
                                                         │
Day 5-7:          Unit 6 Frontend (全员) ─── 依赖全部后端 │
                                                         │
Day 8:            集成测试 + 修复 ────────────────────────┘
```

**总预估**: 8 个工作日（3 人团队并行）
